# Genetic Prompt Optimization for Code Summarization

Applies a genetic algorithm to evolve optimal LLM prompt templates for code summarization. Prompt components (role, task description, guard parameters, formatting instructions) are treated as genes — crossover and mutation explore the space of combinations. Fitness blends lexical overlap (ROUGE-L) and semantic similarity (sentence-embedding cosine) against Opus-generated reference summaries, so the search rewards both surface-level accuracy and semantically-equivalent paraphrases.

## Repo Structure

```
/data
  /functions       # 500 self-contained .py / .js / .ts files (training set the GA samples from)
  /references      # 500 paired .txt summaries generated by Claude Opus (training references)
  /test            # held-out evaluation set the GA never sees during search
    /functions     #   50 distinct functions (20 .py / 15 .js / 15 .ts), zero overlap with /functions
    /references    #   50 paired Opus summaries (same prompt as scripts/generate_references.py)
    /manifest.json
  /component_bank.json  # the GA's gene pool (20 entries per slot × 4 slots = 160k templates)
/src               # GA, RS, fitness, target-model wrapper, embedding/calibration, stats, analysis
/scripts           # CLI entry points (run_experiment, analyze_results, full_eval_and_analyze, plot_convergence)
/results           # per-experiment trial dirs, manifests, summaries, plots (gitignored)
```

## Fitness function

```text
cosine_calibrated = max(0, (cosine_sim − baseline) / (1 − baseline))

fitness(p) = α · ROUGE-L(generated, reference)
           + (1 − α) · cosine_calibrated
           − λ_len · length_penalty(prompt_tokens)
           − λ_fmt · format_penalty(generated)

length_penalty = 0                                      if tokens ≤ max_prompt_tokens
length_penalty = ((tokens − max_prompt_tokens) / max_prompt_tokens) ** exponent   otherwise
```

- `α` (default 0.3): blend weight; cosine carries 0.7 of the signal because semantic match is a superset of the lexical match ROUGE-L provides, while ROUGE-L stays as the lexical anchor to Opus's phrasing.
- ROUGE-L rewards token-level overlap with the Opus reference.
- Cosine similarity uses `BAAI/bge-small-en-v1.5` via `sentence-transformers`. Raw cosine has a non-trivial floor (~0.45) on unrelated text, so it's rescaled against `cosine_baseline` so 0 = "as similar as random unmatched pairs" and 1 = "identical."
- Length penalty has a hard floor at `max_prompt_tokens` (default 200) and grows as `(excess / floor) ** length_penalty_exponent` (default 2.0) — gentle right after the floor, steep far over.
- Format penalty fires on empty output, output containing code fences, or output >10× reference length.

The GA optimizes the blended score directly. Per-metric breakdowns (ROUGE-L only, raw cosine, calibrated cosine, blended) are recorded in `results/` for transparency and ablation.

## How it works

1. GA evolves prompt templates over `data/functions/` using `eval_subset=10` random functions per fitness evaluation (configurable in `config.yaml`). Each trial uses a single seed so candidates within a generation see the same 10 functions.
2. Each candidate is scored against the matching subset of `data/references/` using the blended fitness above.
3. GA is compared against a Random Search baseline at the **same** total evaluation budget (P × G fitness calls) using a Wilcoxon rank-sum test (α = 0.05) on best-of-trial fitness, with Cliff's delta for effect size.
4. After the search, each trial's winning prompt is re-scored on the **held-out test set** (`data/test/`) the GA never sampled from during search — that's the generalization claim.

## Target model

The fixed target model is `nvidia/llama-3.3-nemotron-super-49b-v1.5` served via OpenRouter (`src/target_model.py`, requires `OPENROUTER_API_KEY` in `.env`). It is held FIXED for the duration of any experiment — only the prompt template evolves. The optional directed-mutation improver (`src/improver.py`, gated by `--use-improver`) independently uses NVIDIA NIM and is not part of the core GA-vs-RS comparison.

## Running an experiment

```bash
# 1. Configure (config.yaml — see below for the knobs that matter)
# 2. Kick off the run (caffeinate prevents idle sleep on macOS)
caffeinate -i .venv/bin/python scripts/run_experiment.py

# 3. If anything dies, resume from the most recent experiment dir:
caffeinate -i .venv/bin/python scripts/run_experiment.py --latest

# 4. Re-evaluate the per-trial best prompts on the held-out test set
LATEST=$(ls -1dt results/experiment_*/ | head -1)
.venv/bin/python scripts/full_eval_and_analyze.py \
  --results-root "$LATEST" \
  --output      "$LATEST/test_benchmark_summary.json"

# 5. Optional: legacy training-set re-eval (loud "leaky" warning)
.venv/bin/python scripts/full_eval_and_analyze.py \
  --results-root "$LATEST" \
  --output      "$LATEST/training_set_resample_summary.json" \
  --use-training-set

# 6. Convergence plot (mean ± std bands across trials, GA vs RS, 3 panels)
.venv/bin/python scripts/plot_convergence.py \
  --results-root "$LATEST" \
  --output      "$LATEST/convergence.png"
```

## Validation

There are three distinct evaluation views, each in its own JSON file:

| File | What it measures |
|---|---|
| `statistical_summary.json` (auto, written by `run_experiment.py`) | In-loop fitness on each trial's `eval_subset=10` random functions. Fast in-loop signal — what the GA was actually optimizing against. Noisy by construction. |
| `training_set_resample_summary.json` (`--use-training-set`) | Re-score on all 500 training functions. Removes subsampling noise but stays on the same data the GA searched in — train-set re-eval, not generalization. |
| `test_benchmark_summary.json` (default) | Re-score on the 50 functions in `data/test/` that no trial ever sampled. **The generalization claim.** |

Each file's `evaluation_kind` field labels which kind of run it is. The headline GA-vs-RS comparison should come from the held-out test set; the others are useful for ablation. Statistical validation is the GA-vs-RS Wilcoxon rank-sum on each metric (blended, ROUGE-L, raw cosine, calibrated cosine), with Cliff's delta for effect size — see Romano et al. 2006 thresholds (negligible / small / medium / large) used by `src.stats`.
